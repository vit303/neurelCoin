# DNS Seed Server with Network Crawler

## Общее описание

Данный проект представляет собой реализацию DNS Seed сервера, предназначенного для распределённых сетей, аналогичных Bitcoin P2P. Основная задача системы заключается в сборе, хранении и предоставлении актуального списка активных узлов сети через DNS-запросы. Архитектура решения объединяет в себе асинхронный краулер сети, хранилище пиров и DNS-сервер, обрабатывающий запросы клиентов.

Система автоматически обнаруживает новые узлы сети, периодически обновляет их состояние и предоставляет клиентам выборку доступных адресов в формате стандартных DNS-записей (A, AAAA, SRV).

## Архитектура

Приложение состоит из нескольких ключевых компонентов:

Crawler — асинхронный модуль, отвечающий за обход сети. Он подключается к известным узлам (bootstrap peers), получает от них список других узлов и постепенно расширяет базу данных.

PeerStore — хранилище узлов сети. Управляет добавлением, обновлением и выборкой активных нод. Реализует кэширование для ускорения выдачи результатов.

DNSSeedHandler — обработчик DNS-запросов. Интерпретирует входящие запросы, извлекает условия (например, количество узлов или тип адреса) и формирует корректный DNS-ответ.

UDP DNS Server — сетевой сервер, принимающий DNS-запросы по UDP и передающий их в обработчик.

Protocol Layer — упрощённая реализация взаимодействия с узлами сети по протоколу Bitcoin P2P (в текущем виде содержит заглушку).

## Конфигурация

Основные параметры задаются в классе Config:

* domain — доменное имя DNS seed сервера
* ttl — время жизни DNS-записей
* listen_host — адрес, на котором запускается сервер
* listen_port — порт (в данной версии установлен на 10000)
* crawl_interval — интервал между обходами сети
* max_nodes — максимальное количество хранимых узлов
* bootstrap_peers — начальный список узлов для подключения

Все параметры имеют значения по умолчанию и могут быть изменены перед запуском.

## Принцип работы

После запуска приложения происходит инициализация хранилища узлов и запуск DNS-сервера в отдельном потоке. Затем стартует краулер, который выполняет следующие действия:

Сначала выполняется bootstrap — подключение к заранее заданным узлам сети и получение от них списка пиров.

Далее запускается бесконечный цикл обхода, в котором выбирается случайная подвыборка активных узлов. Для каждого узла выполняется попытка подключения и получения дополнительных адресов.

Полученные узлы добавляются в PeerStore. Если узел уже существует, его данные обновляются.

Параллельно DNS-сервер принимает входящие запросы. При получении запроса:

Проверяется, относится ли домен к обслуживаемому.

Из поддомена извлекаются параметры запроса, такие как:

* тип адресов (IPv4, IPv6)
* количество требуемых узлов
* конкретный node_id

На основе этих условий выбираются подходящие узлы из хранилища.

Формируется DNS-ответ с использованием записей типа A, AAAA или SRV.

## Запуск проекта

Перед запуском необходимо убедиться, что установлен Python версии 3.10 или выше.

Далее требуется установить зависимости. Минимальный набор библиотек:

pip install dnslib bech32

После установки зависимостей необходимо убедиться, что структура проекта соответствует пакетной (наличие **init**.py в директориях).

Запуск осуществляется командой:

python main.py

После старта в консоли появятся сообщения о запуске DNS-сервера и краулера. Сервер начнёт слушать указанный порт (по умолчанию 10000), а краулер — наполнять базу узлов.

## Тестирование

Для проверки работоспособности можно использовать любой DNS-клиент или написать простой тестовый скрипт. Например, можно отправить запрос типа A к домену:

example:

nslookup -type=A yourdomain.com 127.0.0.1 -port=10000

В ответе должны вернуться IP-адреса узлов.

Также можно варьировать параметры запроса через поддомены, например:

* n10.yourdomain.com — запрос 10 узлов
* a1.yourdomain.com — только IPv4
* a2.yourdomain.com — только IPv6

## Особенности реализации

В PeerStore предусмотрено кэширование активных узлов с временем жизни 300 секунд, что снижает нагрузку при частых DNS-запросах.

Для целей тестирования в хранилище добавляются фиктивные узлы, что позволяет проверить работу системы без реального подключения к сети.

Модуль protocol в текущем виде содержит упрощённую логику и имитирует получение узлов. Для использования в production требуется полноценная реализация Bitcoin P2P протокола (обработка version/verack/getaddr/addr сообщений).

## Обработка ошибок

Все сетевые операции обёрнуты в try/except блоки. В случае ошибки DNS-сервер возвращает код SERVFAIL, а краулер игнорирует неудачные подключения.

## Завершение работы

Остановка приложения осуществляется стандартным способом через KeyboardInterrupt (Ctrl+C). При этом выводится сообщение о завершении работы.

## Заключение

Данный проект представляет собой архитектурно корректную и логически завершённую реализацию DNS seed сервера с чётким разделением ответственности между компонентами: DNS-обработчиком, краулером и хранилищем узлов. Решение демонстрирует грамотное использование асинхронного программирования и структур данных, а также соответствует общепринятым подходам, применяемым в системах peer discovery для распределённых сетей.

Вместе с тем необходимо подчеркнуть, что текущая реализация носит прототипный характер. Ключевое ограничение связано с модулем сетевого взаимодействия, в котором отсутствует полноценная реализация P2P-протокола. Функция получения пиров в текущем виде является заглушкой и не обеспечивает получение реальных узлов сети, вследствие чего DNS-сервер фактически оперирует тестовыми данными. Это означает, что система в представленном виде пригодна для демонстрации архитектуры и логики работы, но не может использоваться в production-среде без доработки.

Дополнительно следует отметить, что ряд параметров конфигурации, таких как ограничение максимального количества узлов, в текущей версии не задействованы, а стратегия очистки устаревших данных отсутствует. Также формирование SRV-записей использует доменную зону, не связанную с основным доменом сервиса, что может затруднить корректное разрешение имён клиентами.

Несмотря на указанные ограничения, проект представляет собой качественную основу для дальнейшего развития. При реализации полноценного сетевого протокола, добавлении механизмов логирования, контроля размера хранилища и улучшении генерации DNS-записей система может быть доведена до уровня полноценного DNS seed сервера, пригодного для использования в реальной децентрализованной сети.
